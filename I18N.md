# Lost & Found Korea — i18n Content Strategy

Last updated: 2026-04-28
Supported locales: English (en, default), Japanese (ja)

## 1. Content categories

All content falls into one of four categories:

| Type | Definition | Examples |
|------|-----------|----------|
| 🟰 Shared data | Language-agnostic, defined once | Pricing, categories, facility coordinates, business info |
| 🌐 Both translated | Same message, language-specific phrasing | Hero copy, common FAQs, policy text |
| 🇯🇵 Japan-only | Exists only in /ja | EMS-to-Japan guide, PayPay payment notes |
| 🇺🇸 English-only | Exists only in / | Worldwide shipping map, "no Korean barrier" reassurance |

## 2. Three-layer storage

### Layer 1: Shared data
**Location**: `/config/*.ts`, DB tables, `/public/`

Language-agnostic. Defined once, consumed by all locales.

- Pricing: `/config/pricing.ts` ($39, $79)
- Categories: `/config/categories.ts`
- Facility records (name, address, coordinates, phone, hours): `facilities` DB table
- Business info: `/config/business-info.ts`
- Images: `/public/images/`

### Layer 2: Both translated
**Location**: `/locales/en.json`, `/locales/ja.json`

Same key, different value per locale.

- UI text (buttons, labels, header, footer)
- Marketing copy (hero, about, pricing copy)
- Common FAQ questions and answers
- Policy text (terms, privacy, refund)

Rules:
- Key structure 100% identical between en.json and ja.json
- Array lengths must match (e.g. `pricing.recoveryFeatures` = 4 items in both)
- Placeholders identical (`{amount}`, `{count}`)

### Layer 3: Locale-specific content
**Location**: separate namespaces in locale JSON, plus conditional rendering

#### Pattern A — namespace separation
ja.json:
japanLocale.shippingGuide.* (EMS to Japan)
japanLocale.lostFoundExplainer.* (Korean lost-and-found system explainer)
en.json:
englishLocale.shippingMap.* (worldwide shipping)
englishLocale.supportTrust.* (English-speaking emphasis)

Conditional render in component:
```tsx
{locale === 'ja' && <JapanShippingGuide />}
{locale === 'en' && <WorldwideShippingMap />}
```

#### Pattern B — same array, different IDs
For arrays where most items are shared but some are locale-specific (e.g. FAQ):
en.json: faq.questions = [Q1_id, Q2_id, ENGLISH_ONLY_Q]
ja.json: faq.questions = [Q1_id, Q2_id, JAPAN_ONLY_Q1, JAPAN_ONLY_Q2]

Same `id` = translation of the same question. Different `id` = locale-specific content. Array lengths can differ.

## 4. Decision flow — adding new content

When adding new content, ask 4 questions:

1. **Does it apply to both locales?**
   - YES → Layer 2 (locales JSON)
   - NO → Layer 3 (locale-specific)

2. **Is it data or text?**
   - Data (coordinates, phone, dates) → Layer 1 (DB/config)
   - Text → Layer 2 or Layer 3

3. **Is the meaning identical or are they different messages?**
   - Identical → same key, different values (Layer 2)
   - Different → different keys or separate components (Layer 3)

4. **Will future locales (e.g. zh) reuse this structure?**
   - YES → Layer 1 or 2 (preserve structure)
   - NO → Layer 3 (locale-specific is acceptable)

## 5. Tone guide

### English (en)
- **Audience**: International English speakers (US, EU, SEA, etc.)
- **Positioning**: "English-speaking support, no Korean barrier"
- **Tone**: Friendly, clear, casual is OK
- **Emphasis**: English communication, USD payment, worldwide shipping
- **Example**: "Lost something in Korea? We'll find it."

### Japanese (ja)
- **Audience**: Japanese travelers, Japanese who returned home from Korea
- **Positioning**: "日本語対応・EMS最短2日・PayPal/カード決済"
- **Tone**: Polite (です・ます style), reassuring, trust-emphasizing
- **Emphasis**: Japanese-language support, fast EMS, customs handling
- **Avoid**: Direct English translations, casual tone, lightweight expressions like "簡単!"
- **Example**: "韓国でのお忘れ物、諦めないでください。"

## 7. Workflow for adding new pages or features

1. **Categorize** (which of the 4 types?)
2. **Choose storage layer** (which of 3 layers?)
3. **Write English first** (en.json or component)
4. **Transcreate to Japanese** (ja.json) — request from Claude with context
5. **Review**: visit /ja, check for awkward phrasing
6. **Add SEO meta** (per-locale keywords)
7. **Deploy + request Search Console indexing**

## 8. Hard rules — never do these

- ❌ Use raw machine translation (DeepL/Google) output without review
- ❌ Translate English directly word-for-word into Japanese
- ❌ Leave ja.json with partial Japanese and English placeholders mixed (either fully translate or leave fully English placeholder — never mix)
- ❌ Localize admin UI, backend logs, or internal staff emails (English-only)
- ❌ Display prices in ¥ (always USD for consistency across locales)

## 9. Adding a future locale (e.g. Chinese)

When a new locale is added (e.g. `zh`):

1. Create `/locales/zh.json`, copy structure from en.json
2. Add 'zh' to `SUPPORTED_LOCALES` in `/config/locales.ts`
3. Request transcreation of zh.json from Claude
4. Identify zh-specific sections (e.g. WeChat Pay, 微信)
5. Middleware handles routing automatically (no code changes needed)
6. Create email templates `.zh.tsx`
7. Add zh SEO meta and keywords
8. Submit zh sitemap to Search Console
